# Transm QA Plan

## Scope

This plan assumes users manually provide local audio files for the initial product. Transm should not depend on stream capture, service ripping, or automated source acquisition for v0.x QA. The first quality target is not "studio reconstruction"; it is a conservative, measurable reduction in listening fatigue for Loudness War-era metal and rock without introducing obvious separation, phase, clipping, or tonal artifacts.

The QA process must answer four questions:

1. Does the pipeline run correctly on real local files?
2. Does it avoid damaging audio under conservative defaults?
3. Does it improve the listening experience on suitable crushed masters?
4. Does it detect or warn when source quality or separation quality is too poor to process aggressively?

## Source Policy For QA

Use only files the project is allowed to store and process in the test environment.

Allowed for committed fixtures:

- Synthetic tones, impulses, sweeps, noise bursts, and generated drum-like transients.
- Short fixture files generated by tests at runtime.
- Openly licensed audio with a license that permits redistribution in the repository.
- Derived synthetic degradations of redistributable source audio.

Allowed for local manual QA, but not committed:

- User-owned CD rips.
- Purchased FLAC, ALAC, WAV, or AIFF files.
- Personal working copies of tracks used only for private listening tests.

Not allowed in the repository:

- Commercial tracks.
- Captured streaming-service audio.
- Generated or processed versions of copyrighted commercial tracks.

## Fixture Matrix

Build a local fixture library that separates functional QA from listening QA.

### Committed Mini Fixtures

These should be tiny and deterministic so they can run in unit tests:

- `sine_1khz_stereo.wav`: level, duration, channel, and export sanity.
- `impulse_train.wav`: transient-shaper and limiter behavior.
- `pink_noise_stereo.wav`: EQ and loudness analysis sanity.
- `silence.wav`: NaN, division-by-zero, and quiet-file handling.
- `clipped_sine.wav`: clipping detection and limiter behavior.
- `two_stem_mix.wav`: artificial bass/drum or vocal/guitar split for remix tests.

### Local QA Fixtures

These can live outside git under `qa-fixtures/` and should be documented in a local manifest:

- Clean lossless sources: CD rip or purchased FLAC/WAV.
- Loudness-war-style degraded copies: limiter, clipping, saturation, and high-frequency harshness applied offline.
- Lossy copies: AAC/MP3 at 128, 192, 256, and 320 kbps where available.
- Edge cases: quiet intro, abrupt ending, gapless transition, long silence, dense cymbal section, double-kick section, screamed vocal over guitars.

Each local fixture should have a sidecar manifest:

```json
{
  "id": "local-metalcore-001",
  "source_type": "user_owned_cd_rip",
  "format": "flac",
  "sample_rate_hz": 44100,
  "bit_depth": 16,
  "duration_sec": 213.5,
  "notes": "Dense cymbal chorus, clipped snare, buried bass",
  "may_commit": false
}
```

## Metric Baseline

Every full-pipeline QA run should produce a JSON report with these metrics for input and output:

- Duration.
- Sample rate.
- Channel count.
- Integrated LUFS.
- Loudness range if available.
- Peak-to-loudness ratio.
- Sample peak.
- True peak if implemented.
- Clipped sample count.
- Crest factor.
- Stereo correlation.
- High-frequency energy above 8 kHz and 10 kHz.
- DC offset.
- NaN or infinity count.

Initial acceptance thresholds:

- Output duration must match input within 50 ms for normal processing.
- Output must contain no NaNs or infinities.
- Output must not be silent unless input is silent.
- Output sample peak must not exceed 0 dBFS.
- Final output true peak should target `-1.0 dBTP` once true-peak metering exists.
- Default preset must not increase integrated loudness unless explicitly requested.
- Default preset must not increase high-frequency energy above 8 kHz by more than 1 dB on harsh sources.
- Default preset must not reduce stereo correlation into clearly pathological negative values.

## Functional QA

These tests should run in CI or local `pytest` without external models where possible.

### Audio I/O

- Read WAV and FLAC fixtures.
- Preserve stereo shape.
- Export WAV and FLAC that can be re-opened.
- Reject unsupported formats with actionable errors.
- Handle silence, mono, very short files, and high-amplitude files.

### Analysis

- LUFS and peak metrics are stable for synthetic fixtures.
- Clipping detection flags `clipped_sine.wav`.
- Crest factor changes predictably between sine, noise, and impulse fixtures.
- Reports serialize as deterministic JSON.

### DSP Units

- Transient shaper increases impulse attack without clipping.
- Expander reduces low-level tails without muting desired signal.
- De-esser attenuates a synthetic 7 kHz sibilance band more than broadband content.
- Bass chain does not introduce sub-30 Hz buildup.
- Guitar/fizz chain reduces excessive 8-12 kHz energy on synthetic harsh fixtures.
- Limiter enforces ceiling under expected test conditions.

### Remix

- Summing known stems produces expected shape and duration.
- Stem gain controls are applied in dB accurately.
- Remix headroom management avoids clipping.
- Polarity and latency alignment checks are reported when implemented.

### CLI

- `transm analyze <file>` exits 0 and prints/report metrics.
- `transm process <file> --preset metalcore` writes output and report.
- Bad input exits nonzero.
- Batch processing continues after a single failure and reports failures.

## Separator QA

The separator is likely the largest quality risk. QA must distinguish raw separation quality from DSP quality.

For any source processed through a separator:

- Save raw separated stems.
- Re-sum raw stems before DSP.
- Compare raw re-sum to the input.
- Compute a re-sum residual metric and include it in the report.
- Warn if the raw re-sum is too degraded for aggressive processing.

For open multitrack fixtures with ground-truth stems:

- Compare estimated stems to ground truth where possible.
- Track practical stem bleed indicators, not just SDR-like metrics.
- Check vocal leakage into guitars.
- Check cymbal leakage into vocals.
- Check bass loss from the bass stem.
- Check drum transient smearing.

Separation confidence should produce a traffic-light result:

- `green`: process with selected preset.
- `yellow`: force conservative intensity or warn before aggressive mode.
- `red`: recommend bypass, manual settings, or a different separator backend.

## Source Quality Sensitivity

Do not optimize source acquisition before measuring whether source quality is actually the bottleneck.

For each local QA fixture where a clean source is available, create comparable inputs:

- Lossless FLAC/WAV.
- 320 kbps lossy if available.
- 256 kbps lossy if available.
- 128 kbps lossy if available.
- Loudness-war degraded lossless variant.

Run the same preset across all variants and compare:

- Separation confidence.
- Cymbal artifact level.
- Vocal warble.
- PLR and LUFS deltas.
- High-frequency energy changes.
- Listener preference.

Decision rules:

- If 320 kbps and lossless are hard to distinguish after processing, do not spend v0.x engineering time on source acquisition.
- If 128 kbps consistently creates obvious artifacts, add a low-quality source warning.
- If lossy sources consistently worsen cymbals, route them to a conservative preset or disable aggressive transient enhancement.

## Listening QA

Metrics are guardrails, not proof of quality. Every release candidate should include structured listening tests.

For each candidate preset, export:

- Original input.
- Conservative output.
- Default output.
- Aggressive output.
- Raw re-sum before DSP if a separator was used.

Use blind or randomized filenames when practical. Rate each file from 1 to 5:

- Punch.
- Bass clarity.
- Cymbal fatigue.
- Vocal naturalness.
- Guitar body.
- Artifact level.
- Overall preference.

Listening checklist:

- Are cymbals less fatiguing, or just duller?
- Did the snare get punchier, or just clickier?
- Did guitars lose body?
- Did bass become audible without adding mud?
- Did vocals become recessed, warbly, or phasey?
- Are there watery artifacts in quiet passages?
- Is the breakdown more impactful at equal loudness?
- Would a full album listen be easier than the original?

Default preset release gate:

- No obvious clipping.
- No obvious vocal warble on the local fixture set.
- No obvious cymbal exaggeration.
- At least 60 percent blind preference over input on suitable crushed fixtures.
- Aggressive mode remains opt-in.

## Release Milestones

### v0.1: Pipeline Safety

- Audio I/O works on committed fixtures.
- Analysis report is generated.
- Conservative process command exports a valid file.
- Output duration and channels are correct.
- No clipping, NaNs, or silent-output regressions.
- Unit tests pass without requiring heavyweight separation models.

### v0.2: Controlled Improvement

- Add local QA fixture manifest support.
- Add source-quality sensitivity runs.
- Add raw stem re-sum report.
- Add first listening scorecard template.
- Tune `metalcore` preset against local fixtures.

### v0.3: Quality Gating

- Add separation confidence.
- Add warnings for poor or lossy sources.
- Add preset acceptance thresholds.
- Add QA report comparison across commits.
- Add blind A/B export helper.

### v1.0: Release Confidence

- Stable default preset on a meaningful local fixture set.
- Documented known failure modes.
- Cross-platform processing tests.
- Reproducible QA reports.
- No source-acquisition dependency beyond user-provided local files.

## Manual QA Command Shape

The final commands may change, but the workflow should converge on this shape:

```bash
transm analyze input.flac --json reports/input.json
transm process input.flac --preset metalcore --intensity 0.35 --output out.flac --report reports/out.json
transm qa compare input.flac out.flac --json reports/compare.json
transm qa abx input.flac out.flac --output qa-abx/
```

Batch local QA should look like:

```bash
transm qa run qa-fixtures/manifest.json --preset metalcore --output qa-results/latest/
```

## Non-Goals For Initial QA

- Proving perfect reconstruction of clipped transients.
- Proving the output is equivalent to an original multitrack remix.
- Optimizing for hi-res sample rates above 44.1 or 48 kHz.
- Building or validating streaming-service capture.
- Training a custom restoration model.
- Shipping a GUI before the CLI pipeline is stable.
